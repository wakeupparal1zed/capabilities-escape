# capabilities-escape

## Что проверяет таск

Таск на злоупотребление Linux capabilities вместо классического SUID. Игрок должен понять, что привилегия выдана не через setuid-бит и не через sudo, а через capability на исполняемом файле.

## Какая здесь уязвимость

В контейнере интерпретатор `python3` получает capability `cap_setuid+ep`. Это означает, что процесс, запущенный через этот бинарь, может успешно вызвать `setuid(0)` и сменить эффективный UID на `root`.

Проблема в том, что capability выдана универсальному интерпретатору. Для прикладного бинаря с жестко зафиксированной логикой это еще можно было бы как-то оправдывать, но для Python это фактически выдача примитива на выполнение произвольного кода с возможностью стать `root`.

## Как уязвимость возникает в этой сборке

В [`entrypoint.sh`](./entrypoint.sh) capability назначается на реальный путь к системному `python3` через `setcap cap_setuid+ep`. Одновременно из контейнера убираются другие простые пути эскалации: режутся SUID-биты и блокируется `su`. За счет этого intended path остается чистым и завязанным именно на capabilities.

## Почему это работает

Linux capabilities дробят root-права на отдельные привилегии. `CAP_SETUID` разрешает процессу менять свой UID без обычных ограничений. Если capability находится в effective set процесса, код внутри Python может вызвать системный `setuid(0)`, после чего дальнейшие действия уже выполняются с root-привилегиями.

Критично именно то, что capability висит на интерпретаторе общего назначения. Игроку не нужно ломать сам Python или искать memory corruption: достаточно запустить собственный код внутри уже привилегированного процесса.

## Почему это хороший учебный сценарий

Таск показывает частую ошибку конфигурации: администратор убирает SUID и sudo, но выдает capability слишком мощному бинарю, считая это более безопасной альтернативой. На практике такая настройка все равно приводит к полной компрометации контейнера.
